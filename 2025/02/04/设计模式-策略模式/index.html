设计模式-策略模式

引言：

在日常生活中常常会遇到为了实现某种目标存在多个方案可以选择的情况，例如：去三亚旅游，是坐飞机，还是做火车还是坐游轮？在商店购买商品的时候采用的支付方式是现金，支付宝，还是微信等等。其实在我们的软件开发过程中也会有类似的情况，下面就由我来介绍一下今天的主角：策略模式。

正文：

何为策略模式？

策略模式属于设计模式中容易理解的一种模式，这种属于行为型设计模式，主要目的是定义一系列算法，把这些算法一个个的封装起来，并且使它们可以互相替换。这样的算法变化就可以独立于客户端之间了。

Java 标准库中也不乏策略模式的身影，最常见的就是 Comparator 接口：

Collections.sort() 方法接收不同的 Comparator 实现，这就是典型的策略模式——算法（排序规则）与调用方（排序方法）解耦。

策略模式的优缺点

优点
  1. 开闭原则：新增支付方式（如银行卡支付）时，只需新增一个实现类，无需修改现有代码。
  2. 消除条件判断：避免了大量的 if-else 或 switch-case 语句，代码更清晰。
  3. 策略可复用：同样的策略可以在多个场景下复用。
  4. 运行时切换：可以在程序运行过程中动态替换算法/行为。
缺点
  1. 类膨胀：每增加一个策略就需要新增一个类，策略较多时类数量会激增。
  2. 客户端必须了解策略差异：客户端需要知道不同策略之间的区别，才能选择合适的策略。
  3. 策略之间相互独立：策略模式默认策略之间是独立且不可互相调用的。

适用场景

  • 系统需要在多种算法/行为中动态选择一种
  • 需要避免使用多重的条件判断语句
  • 算法/行为需要被客户端完全独立使用
  • 算法的实现细节需要对客户端隐藏

总结

策略模式是一种行为型设计模式，它将算法与调用方解耦，让算法可以独立于使用它的客户端而变化。通过定义统一的策略接口，将不同的算法封装成独立的策略类，客户端可以在运行时灵活切换。

回到开头的例子：去三亚旅游选择交通工具，本质上也是一种策略选择。策略模式让我们的代码像出行一样灵活——今天想坐飞机就坐飞机，明天想坐火车就坐火车，不需要改变旅行计划本身，只需要更换”策略”即可。
